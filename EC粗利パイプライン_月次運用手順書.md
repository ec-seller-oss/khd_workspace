## 全体像（2レーンで「実額粗利」を出す）

```
EC粗利(円) = 【①売上レーン】実精算額(円) − 【②仕入レーン】仕入原価(円)

①売上レーン  ：クーパンが精算した実額 → 販売管理表(見込み)と照合
②仕入レーン  ：Amazon仕入履歴 → MF仕訳帳に合わせて集計
```

**ポイント：** ダッシュボードの粗利は「見込み（手数料11%固定）」ベース。実際の手数料は10.5〜10.8%なので、実精算は+1.65%多い（実態はダッシュボードより¥18万健全）。

### ステップ3: 精算実額照合 ★月次で必ず回す

```bash
python3 scripts/ec_settlement_recon_v2.py
```

**入力:**
- `韓国輸出売上/_精算実額MSF/MSF_PAYMENT_REVENUE_DETAIL*.xlsx`（ゆーしがDLしたファイル群）
- `韓国輸出売上/250811_売掛金.xlsx`（前半タレ・変更不要）

**出力:** `ec_settlement_recon_v2.csv`（796行・照合明細）

**必ず確認する3点:**
```
① カバレッジ → 各月90%以上あるか（❌の月はゆーしにその月のDLを依頼）
② A.売上漏れ候補 → 件数と金額を確認。あれば販売管理表のstatusを「発送済」に修正
③ B.キャンセル補償 → 何もしない（雑収入で正しい）
```

**フラグの意味:**

| 区分 | 対応 |
|------|------|
| 照合OK | 問題なし |
| 精算待ち（実精算なし） | 精算未到来。次月以降のDLで解消 |
| A_売上漏れ候補（status要修正） | 販売管理表のstatusを「発送済」に修正して売上計上 |
| B_キャンセル補償（雑収入） | 何もしない。売上ではなく雑収入 |

## 精算待ち・欠落の対処

```
カバレッジが低い月があった場合のチェックフロー:

1. ec_settlement_recon_v2.py の「月別カバレッジ」で❌の月を確認
2. その月の精算予定日（例: 2026-01-22）はいつか確認
3. ゆーしに「〇月分のDLをお願い」→ _精算実額MSFに追加してもらう
4. スクリプト再実行 → ❌が🟡か✅に変わるか確認
```

**2025年12月の38件未収録（既知）:** クーパン2の精算予定日 2026-04-21 / 05-26 のバッチが未DL。
ゆーしに追加DL依頼すればほぼ100%に到達。

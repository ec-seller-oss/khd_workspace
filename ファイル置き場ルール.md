# ファイル置き場ルール（迷子ゼロの判断基準）

作成: 2026-07-10 フォルダ全面監査にて（復元ログ: `.company/secretary/notes/2026-07-10-folder-restructure-log.md`）

## 3秒判断フロー（上から順に当てはめる）

1. **自動化が読み書きするデータか？**（EC pipeline のcsv/db等）→ **ルート直下**（動かすと壊れる。凍結リスト参照）
2. **スクリプト（道具）か？** → `tools/`
   - スライド生成 → `tools/slides/`／シート・データ処理 → `tools/sheets/`／GAS → `tools/gas/`
   - ※常駐自動化（hook・launchd・CLI）だけは `scripts/`（絶対パス参照があるため不可侵）
3. **完成した配布物・資料か？** → `成果物/<本部>/`
   - `00_経営_マニュアル`／`00_調査士学習`／`02_資金税務`／`03_EC`／`04_医療`／`05_物件`
   - 同一案件で3点以上溜まったら `成果物/<本部>/<案件名>/` を掘る（例: 04_医療/京橋クリニック、05_物件/辻堂東海岸）
4. **秘書室の記録（メモ・決定・学び・日報）か？** → `.company/secretary/`（既存ルール通り）
5. **独立したアプリ/プロジェクトか？** → 専用フォルダ（landscanner等）。乱立させない、新設は要相談
6. **迷ったら** → `_保留/` に入れて秘書に一言（勝手に捨てない・勝手に新フォルダを作らない）

## 原則

- **階層は最大3段**（成果物/本部/案件 まで）。それ以上掘らない — 検索はAI（秘書）とGitHub全文検索が担う
- **生成スクリプトの出力先は `成果物/<該当本部>/`**（旧「フォルダ直下に出力」ルールは2026-07-10廃止）
- 削除はしない。役目を終えたら `_保留/` へ
- 命名: 日付付き `YYMMDD_` or `_YYYYMMDD` を推奨（既存慣習に合わせる）
- ルート直下に新規ファイルを置いてよいのは「凍結リスト該当（自動化データ）」のみ

## 復元方法

全移動は `git mv`（履歴保持）。個別復元は復元ログの右→左へ `git mv`、全戻しは該当コミットを `git revert`。
